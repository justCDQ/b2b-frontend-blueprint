# Toast / Feedback 反馈组件规范

> 用于创建、修改或审查 2B 产品中的反馈提示。
> 反馈应该让用户知道结果，而不是制造噪音。

---

## 1. 组件用途

Toast 用于用户操作后的短反馈，或重要异步结果提醒。

Toast 不用于：

- 普通数据加载成功。
- 替代字段错误。
- 展示长内容。
- 解释复杂业务状态。
- 频繁轮询更新。

---

## 2. 什么时候弹 Toast

用户主动操作后可以弹：

- 创建成功/失败。
- 编辑成功/失败。
- 删除成功/失败。
- 启用/禁用成功/失败。
- 复制成功/失败。
- 导入/导出任务已创建。
- 批量操作汇总结果。
- 用户操作后权限不足。
- 操作被业务规则阻止。

重要异步事件可以弹：

- 后台任务完成。
- 支付/回调结果。
- 长时间导入/导出完成。
- 用户可处理的同步失败。

示例：

```text
Token 已创建
用户已禁用
已复制到剪贴板
已删除 12 项
导出任务已创建
```

---

## 3. 什么时候不弹 Toast

不要为这些预期行为弹 toast：

- 初始数据加载成功。
- 普通刷新成功。
- 轮询成功。
- 静默刷新失败。
- 请求被取消。
- 过期响应返回。
- 页面跳转成功。
- 登录成功且已经跳转。
- 字段级校验错误。
- 批量操作中的每一项结果。

替代方式：

| 场景 | 使用 |
|---|---|
| 字段错误 | inline field error |
| 表单级校验 | form-level error |
| 空数据 | EmptyState |
| 初次加载错误 | ErrorState + retry |
| 长解释 | Dialog / Panel |
| 系统级警告 | Banner |

禁止：

```text
数据加载成功。
刷新成功。
轮询每 10 秒失败弹一次。
批量删除每条数据弹一次。
```

---

## 4. Toast 类型

| 类型 | 使用场景 |
|---|---|
| success | 用户操作完成 |
| error | 用户操作失败 |
| warning | 操作被阻止、部分成功、风险状态 |
| info | 中性异步提醒或任务已创建 |
| loading | 短时间进行中的操作，且确实有必要 |

规则：

- success 文案要短。
- error 文案要说明哪里失败。
- warning 文案要说明用户下一步能做什么。
- info 不用于常规噪音。
- loading toast 要最终变成 success/error 或消失。

---

## 5. 频率与去重

Toast 不能噪音堆叠。

规则：

- 相同错误在短时间内去重，例如 2 秒。
- 批量操作只产生一条汇总 toast。
- 轮询重复错误不弹 toast。
- 快速重复点击不能产生重复 toast。
- 操作 pending 时禁用触发器，而不是允许重复点击再弹 toast。

批量示例：

推荐：

```text
已删除 12 项
删除完成：成功 9 项，失败 3 项
```

不推荐：

```text
已删除 A
已删除 B
已删除 C
```

---

## 6. 弹出位置

桌面端默认位置：

- 右下角。

右下角适合：

- CRUD 成功/失败。
- 复制结果。
- 批量操作汇总。
- 导入/导出任务创建。
- 非阻塞操作结果。

页面中间适合：

- 初次页面加载。
- 阻塞型页面错误。
- 空状态。
- 无权限空状态。
- 整页不可用状态。

这些情况用页面内反馈，不用 toast：

- 表单字段错误。
- table 初次加载错误。
- 筛选后无结果。
- 区块级错误。

左下角只在这些情况使用：

- 产品右侧有常驻面板，会遮挡 toast。
- 右侧用于详情抽屉或聊天面板。
- 产品标准明确选择左下角。

规则：

- 全应用应该有一个默认 toast 位置。
- 不要没有布局原因就每个页面换位置。

---

## 7. 持续时间

推荐持续时间：

| 类型 | 时间 |
|---|---|
| success | 1500-2500ms |
| info | 3000ms |
| warning | 5000-8000ms |
| error | 5000-8000ms |
| loading | 持续到 resolved 或最大超时 |

规则：

- 破坏性操作失败停留时间应比 success 更长。
- 带 action 的 toast 要给用户足够点击时间。
- 长文本不要塞进 toast，使用 dialog/panel。
- 用户应该可以手动关闭 toast。

---

## 8. 移动端行为

移动端 toast 不能遮挡关键操作。

规则：

- 如果不遮挡底部导航/操作，优先底部展示。
- 如果底部有关键操作，使用顶部或安全区偏移。
- Toast 宽度适配移动端视口。
- 文案最多建议 2 行。
- 移动端避免多个 toast 堆叠。
- 批量/长反馈内容较多时，用 dialog 或结果页。
- 移动端不能依赖 hover 解释，必要时使用点击详情。

移动端持续时间：

- success 可以更短。
- error/warning 要保留足够阅读时间。

---

## 9. Toast 文案

规则：

- 文案短。
- 文案必须和动作相关。
- 避免“操作成功”这类模糊文案。
- 批量操作包含数量。
- 对象名称只在有帮助时出现。
- 不展示 secret、token 或敏感数据。
- 不展示原始 stack trace。

推荐：

```text
Token 已创建
API Key 已复制
已禁用用户 "alice@example.com"
导入完成：成功 120 项，失败 3 项
```

不推荐：

```text
成功
错误
发生了一些问题
eyJhbGciOi...
```

---

## 10. 带操作的 Toast

只有操作真的有用时才加 action。

示例：

- 查看任务。
- 重试。
- 下载错误报告。
- 撤销安全可逆操作。

规则：

- action 文案要短。
- action 行为必须安全且可预期。
- 不要在 toast 里承载复杂决策。
- 危险确认不应该发生在 toast 里。

---

## 11. AI 审查清单

接受反馈/Toast 代码前，检查：

- Toast 是否只用于合适的用户操作反馈。
- 普通加载/刷新/轮询成功是否没有 toast。
- 取消或过期请求是否没有 toast。
- 字段错误是否使用 inline error，而不是 toast。
- 批量操作是否只用一条汇总 toast。
- 重复错误是否去重。
- 位置是否遵循应用默认，除非布局有原因。
- 页面中间反馈是否用于阻塞 loading/error/empty。
- 持续时间是否匹配严重程度。
- 移动端 toast 是否不遮挡关键操作。
- Toast 文案是否短且动作明确。
- Toast 中是否没有敏感数据。
